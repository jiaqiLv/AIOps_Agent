## 执行说明

你需要基于**实际执行的算法结果**进行精化。并非所有算法都可能被执行：
- 如果工具调用失败，对应结果可能不可用
- 如果参数不足，某些算法可能被跳过
- 综合所有可用结果，给出尽可能完整的分析

## 输入数据说明

### 可能的输入组合

| 执行的算法 | 说明 | 处理方式 |
|---|---|---|
| RCD + PC | 提供了 inject_time 和 abnormal_kpi | 综合两个算法的结果 |
| 仅 PC | 只提供了 abnormal_kpi 或只有 csv_path | 基于PC结果分析 |
| CSV失败 | 无数据 | 返回错误 |

### RCD 输出（如果执行了）
- **root_causes**: 候选根因指标列表
- **abnormal_kpi**: 异常KPI名称（如果提供）
- **特点**: 排名越靠前，越可能是根因
- **依据**: 基于故障注入时间前后的变化

### PC 输出（总是执行）
- **root_causes**: 候选根因指标列表
- **causal_graph**: 因果图结构
- **edges**: 有向边（表示因果关系）
- **abnormal_kpi**: 异常KPI名称（如果提供）
- **特点**: 基于条件独立性测试，如果提供abnormal_kpi会找出影响该指标的根因

## 精化任务

### 1. 根因指标精化

**合并策略**：
- RCD 和 PC 都命中的指标 → 高置信度，标注 "RCD+PC"
- 只有 RCD 命中的指标 → 中置信度，标注 "RCD"
- 只有 PC 命中的指标 → 低置信度，标注 "PC"
- PC 中异常KPI的祖先节点 → 优先考虑

**过滤规则**：
- 指标名称与故障类型无关 → 删除并说明原因
- 指标与异常 KPI 无因果路径 → 删除并说明原因
- 不合理的因果关系（如 error → cpu）→ 标注并谨慎处理

### 2. 传播图精化（使用 PC 结果）

**边方向验证**：
- CPU usage → latency ✓
- Error rate → CPU usage ✗（反向，不合理）
- Latency → error rate ✓

**路径提取**：
- 如果有 abnormal_kpi: 找出所有到 abnormal_kpi 的路径
- 如果没有 abnormal_kpi: 找出所有从根因出发的路径

### 3. 故障类型判断

基于根因指标类型判断：

| 根因指标 | 故障类型 |
|---|---|
| *_cpu_usage, *_cpu_util | CPU_RESOURCE_SATURATION |
| *_memory_usage, *_mem | MEMORY_LEAK |
| *_latency, *_rt | LATENCY_SPIKE |
| *_error_rate, *_errors | ERROR_SURGE |
| *_disk_io, *_iowait | DISK_BOTTLENECK |
| *_network, *_packet_loss | NETWORK_ANOMALY |
| db_*_connection, db_*_query | DATABASE_BOTTLENECK |

### 4. 置信度评估

| 执行情况 | 置信度 |
|---|---|
| RCD 和 PC 高度一致，传播路径清晰 | 0.8-1.0 |
| RCD 和 PC 部分一致，传播路径基本合理 | 0.6-0.8 |
| 只有 PC 执行，结果合理 | 0.4-0.6 |
| 只有 RCD 执行，结果合理 | 0.4-0.6 |
| 算法结果不明确 | 0.2-0.4 |
| 所有算法失败 | 0.0-0.2 |

## 注意事项

1. **基于实际输出**: 只精化实际执行的算法结果
2. **不要编造**: 不要添加算法未输出的核心根因
3. **解释原因**: 对每个过滤操作给出理由
4. **保持客观**: 基于数据说话，避免臆测
5. **明确限制**: 说明哪些信息因为参数不足而无法获得
6. **标注来源**: 明确每个根因指标来自哪个算法
